# Decision Log: Agentic Market Strategist PRD

This log tracks all architectural decisions, assumptions, and revisions made during the design and lifecycle of the Agentic Market Strategist PRD.

---

## [2026-06-06] - Initial Setup & Scope Alignment

### DEC-001: Language Standard Execution
- **Decision:** The PRD, decision logs, and all downstream artifacts will be written in **English** instead of French (overriding the default config value for this run).
- **Rationale:** Ensures perfect continuity with the Java codebase, parser, and agent rules specified in `AGENTS.md`.

### DEC-002: Module Integration Vector
- **Decision:** The orchestration layer will be integrated into the existing Maven module: `trading-intelligence`.
- **Rationale:** Avoids codebase fragmentation and module duplication, since `trading-intelligence` already contains LLM-related clients, jobs, and planners.

### DEC-003: API Data Harmonization
- **Decision:** The LLM tools will reuse existing types (e.g., `com.martinfou.trading.data.EconomicCalendar`) and guarantee UTC `Instant` timeline compliance.
- **Rationale:** Prevents duplicate models and aligns with the Epic 2-10 timezone migration rules.

---

## [2026-06-06] - Validation Adjustments & Hardening

### DEC-004: Segregation of Downstream DTOs
- **Decision:** Define all shared target DTO records (like `WeeklyStrategyOutlook` and `TradeTriggerCondition`) in `trading-core`.
- **Rationale:** Keeps the dependency graph acyclic. Higher-level modules (like `trading-intelligence`) must not be depended upon by downstream strategy execution modules.

### DEC-005: Prevent Lookahead Bias in Sentiment & Seasonality Tools
- **Decision:** Add an `Instant timestamp` parameter to the tool methods for both `fetchMarketSentiment` and `fetchWeeklySeasonality`.
- **Rationale:** Guarantees that the LLM queries historical sentiment states and seasonality performance states corresponding to the backtest timeline, preventing lookahead bias.

### DEC-006: Elimination of Redundant Data Types
- **Decision:** Delete the custom `TriggerType` enum and the redundant `CalendarEvent` record. Instead, reuse `com.martinfou.trading.core.Order.Side`, `com.martinfou.trading.core.Order.Type`, and `com.martinfou.trading.data.EconomicCalendar.Event`.
- **Rationale:** Adheres strictly to the DRY principle and avoids needless class bloat and serialization overhead.

### DEC-007: Shifting Mathematics to Programmatic Java
- **Decision:** Remove fractional math calculations (e.g. 2.5x multipliers and 5-15% probability down-adjustments) from the LLM system prompt constraints.
- **Rationale:** LLMs are notoriously non-deterministic and unreliable for direct fractional arithmetic. These mathematical weighting adjustments will be computed programmatically in Java after extracting raw metrics from the LLM.

### DEC-008: Default Model Specification
- **Decision:** Set `gpt-4o` (or `deepseek-chat` / configurable alternative) as the default model provider configuration for LangChain4j.
- **Rationale:** Establishes a baseline for system prompt tuning, testing, cost estimations, and capability validation.

---

## [2026-06-06] - Final Hardening & Schema Segregation

### DEC-009: Injection of Current Price Context
- **Decision:** Programmatically inject the current close price of the asset at the execution timestamp into the LLM prompt context as a variable (`currentAssetPrice`).
- **Rationale:** Provides the LLM with immediate price reference context so it can generate mathematically consistent trade setups, avoiding validation failures.

### DEC-010: Separation of Raw LLM Schema from Final DTO
- **Decision:** Define `WeeklyStrategyOutlookRaw` inside `trading-intelligence` for raw LLM JSON serialization (excluding `comfortLevel`). The Java service layer will map it to `WeeklyStrategyOutlook` (defined in `trading-core`) after computing the comfort level programmatically.
- **Rationale:** Keeps the DTO mappings clean and prevents LangChain4j from forcing the LLM to output fields that are designated for programmatic Java computation.

### DEC-011: Telemetry and Logging for Deterministic Fallback
- **Decision:** Enforce logging via SLF4J and increment a programmatic fallback failure metric when the deterministic bypass triggers.
- **Rationale:** Prevents silent degraded failures of the LLM integration under schema drift or rate limits.

---

## [2026-06-06] - Final Validation Refinements

### DEC-012: Prevents Lookahead in Economic Calendar Tool
- **Decision:** Enforce that the calendar tool wrapper must clear the `actual` value field for any macroeconomic event with a timestamp greater than the current simulation/evaluation timeline.
- **Rationale:** Stops the LLM from seeing future macroeconomic numbers in historical backtests.

### DEC-013: Exhaustive Comfort Level Decision Logic
- **Decision:** Formulate an exhaustive three-tier decision logic rule in Java to compute ComfortLevel, mapping all possible state permutations cleanly.
- **Rationale:** Prevents null outputs or exceptions when converting raw LLM strategy outputs into the final domain DTOs.
